# /interview — взять интервью у владельца

Этот навык фиксирует решения, которые **нельзя принимать автономно**, в md-документ в
`interviews/` и приостанавливает работу, пока владелец не ответит.

Все интервью живут в `interviews/interview_NNN_<topic>.md`.

## Когда вызывать (это РЕДКОЕ событие)

По умолчанию агент работает **автономно** и принимает технические решения с разумными значениями по умолчанию.
Интервью — исключение. Вызывай его ТОЛЬКО когда вопрос действительно владельца:

- **Решение по UI/UX** — как что выглядит, ведёт себя, ощущается для конечного пользователя. Никогда не делай выбор по UI/UX
  без подтверждения.
- **Серьёзная техническая развилка** — выбор библиотеки/протокола/архитектурного подхода с долгоживущими,
  трудно обратимыми последствиями.
- **Бренд / видение / приоритеты продукта** — нейминг, иконка, целевые платформы, что входит в фазу, а что нет.

НЕ вызывай интервью для: мелких деталей реализации, имён переменных, обычных исправлений багов,
рефакторингов, выбора между равнозначными техническими вариантами — это решай сам и отчитайся в чате.

Если не уверен «это уровень владельца или мой?», спроси: *дёшево ли это откатить?* Если да — решай сам.
Если это надолго формирует бренд/архитектуру/UX — проводи интервью.

## Процедура

### Шаг 1. Подготовка (до написания вопросов)
- Прочитай контекст в коде/документах — не спрашивай то, что можешь узнать сам.
- Проверь технические факты, определяющие, какие варианты вообще возможны (например, «можно ли убрать этот
  диалог?», «есть ли у библиотеки нужный API?»). Вопрос без проверенной подготовки — плохой вопрос.
- Посмотри прошлые интервью (`ls interviews/`), чтобы не дублировать принятые решения и держать единый стиль.

### Шаг 2. Создай документ интервью
- Имя: `interviews/interview_NNN_<short_topic>.md`, где `NNN` — следующий свободный номер
  (`ls interviews/` → макс+1, формат `004`, `005`, …).
- Шаблон:
  ```markdown
  # Interview #NNN — <Тема>

  > Topic: <одно предложение о том, про что это интервью>
  > Source of the idea: <файл/чат, дата>
  > Status: **🟡 awaiting the owner's answers**

  ## Context / what I already found in the code
  <кратко: текущее состояние + проверенные технические факты, ограничивающие варианты>

  ## QUESTIONS

  ### Q1. <вопрос>
  - **A) (recommended)** <вариант + почему>
  - **B)** <вариант>
  - **C)** <вариант, если есть>

  **Answer:**

  ### Q2. ...

  ## Proposed implementation plan (after answers)
  <шаги, которые ты предпримешь после закрытия вопросов>
  ```

### Шаг 3. Правила хороших вопросов
- **Закрытые** варианты A/B/C — а не открытое «как лучше?».
- **Рекомендация первой**, помеченная `(recommended)` — у владельца всегда есть быстрый дефолт.
- Каждый вариант с коротким «почему» / компромиссом.
- Группируй: 1–5 вопросов на интервью, не больше.
- Не спрашивай то, что уже решено в `plans/` или прошлых интервью.

### Шаг 4. Спроси владельца — через документ
Дефолтный, дружелюбный к автономии способ: владелец отвечает **прямо в md-документе** (заполняет
поля «**Answer:**»). Это держит работу асинхронной — агент не заблокирован на синхронном чате.

Последовательность:
- Составь `interviews/interview_NNN_<topic>.md` с вопросами и полями «**Answer:**».
- Напиши ОДИН абзац в чат: что нашёл, какие развилки, и ссылку на документ.
- **Приостанови** работу (чтобы владельцу подали сигнал прийти и заполнить ответы). Не угадывай за него и
  не продолжай вслепую по вопросам UI/UX/бренда/архитектуры.

### Шаг 5. После ответов
- Запиши решения в документ: смени статус на `✅ ANSWERS RECEIVED <date>` и добавь
  таблицу «Decisions».
- Если решения затрагивают план/архитектуру — обнови `STATUS.md` и соответствующие файлы в `plans/`.
- Приступай к реализации по одобренному плану (или, если владелец просит сделать паузу — вызови `/pause`).

## Заметки
- Стиль и язык — под владельца.
- Прошлые интервью — эталон для тона и структуры.
- Цель навыка — минимизировать беспокойство владельца, но НЕ принимать за него судьбоносные решения.
