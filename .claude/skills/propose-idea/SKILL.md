# /propose-idea — оформить идею в plans/ideas/ (на одобрение владельца)

Когда у агента появляется стоящая идея (фича, улучшение UX/архитектуры, инструмент), он оформляет её
ОТДЕЛЬНЫМ md в `plans/ideas/` по тем же правилам, что и существующие идеи. **Идея агента — это вклад
в ВИДЕНИЕ продукта, а видение принадлежит владельцу (человеку).** Поэтому такая идея создаётся со
статусом «ожидает одобрения» и **НЕ реализуется, пока владелец её не одобрит**.

## Когда вызывать

- Возникает идея, которая двигает проект к целям из `master_plan.md`/`PHILOSOPHY.md` и не
  противоречит видению владельца.
- НЕ для мелких технических решений внутри уже одобренной задачи (их решай сам и просто делай).
- НЕ для вопросов, требующих ВВОДА владельца по уже идущей работе — это `/interview` (вопрос), а не
  идея. Разница: `/interview` = «мне нужен твой ответ, чтобы продолжить»;
  `/propose-idea` = «я предлагаю НОВОЕ направление, оцени его».

## Что делать

1. **Сверься с видением ПЕРЕД оформлением.** Прочитай (или вспомни) `plans/master_plan.md`, `PHILOSOPHY.md`
   (KISS) и решения в `interviews/`. Идея должна согласовываться. Если она противоречит / уводит в сторону /
   усложняет без необходимости — не предлагай её (или переформулируй проще).

2. **Определи следующий номер.** `ls plans/ideas/` → максимальный `NN` + 1. Имя:
   `plans/ideas/NN_<short_english_name>.md` (как у соседей; БЕЗ тега `DONE`).

3. **Напиши документ по канону** (структура как у существующих идей):
   ```
   # Idea NN — <название>

   > Source: AI agent (dayloop/nightloop/observation), <date>.
   > Status: ❓ AWAITING HUMAN APPROVAL — идея агента = элемент видения; не реализуется без одобрения.

   ## Essence
   <что предлагается, кратко>

   ## Why / what problem it solves
   <боль/возможность; для кого>

   ## How it fits the master plan and vision
   <чёткая привязка к master_plan / принятым решениям — почему это «в русле», а не вбок>

   ## Implementation sketch (KISS)
   <грубый подход; переиспользование того, что уже есть; объём/риски>

   ## Open questions / what needs the human's answer
   <развилки, которые должен закрыть владелец>
   ```

4. **Отметь на ревью и НЕ реализуй:**
   - Строка в `STATUS.md`: «❓ ожидает ревью владельца: идея NN — <одна строка>».
   - Закоммить документ (`node tools/commit.mjs "docs(ideaNN): proposal — …"`).
   - **НЕ начинай реализацию**, пока владелец явно не одобрит. В автоцикле — продолжай ДРУГИМИ задачами.

5. **После реакции владельца:**
   - Одобрено → бери в работу (технические решения внутри — твои). После реализации: статус ✅ +
     дата, и по соглашению о теге DONE — `git mv` с тегом `DONE` и разделом статуса внутри.
   - Отклонено/переформулировано → отрази его решение в документе (или удали идею, если отклонена).

## Заметки
- Идея — это ПРЕДЛОЖЕНИЕ, а не разрешение действовать. Дисциплина: видение продукта формирует владелец.
- Лучше осмысленно собрать несколько небольших идей, чем плодить почти-дубликаты. Проверь существующие открытые
  идеи (`ls plans/ideas/`), чтобы не дублировать.
