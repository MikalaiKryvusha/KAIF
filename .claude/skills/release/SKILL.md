# /release — опубликовать релиз на GitHub

Владелец просит выпустить новую версию. Это **необратимое внешнее действие** (публичный тег +
GitHub Release). Выполни процедуру **по порядку**; проговаривай каждый шаг в чате. Если шаг падает — остановись,
покажи ошибку, НЕ продолжай вслепую.

> ⚠️ **ТРЕБУЕТСЯ ПОДТВЕРЖДЕНИЕ.** Перед самой публикацией покажи владельцу: какая будет версия
> (текущая → новая), что дерево чистое, что собралось. Публикуй только по его явному «да». Релиз
> = публичный тег и Release, откатывать неприятно. **В автономном режиме (`/autoloop`/циклы) НЕ
> публикуй релиз.**

## Шаг 0. Определи тип поднятия версии

Подтверди с владельцем (или подтверди дефолт): patch / minor / major. Назови текущую → новую версию.

## Шаг 1. Предпроверка окружения (не релизь на грязном/сломанном дереве)

```bash
git status --short          # дерево должно быть ЧИСТЫМ (кроме gitignore-артефактов)
git branch --show-current   # релизная ветка (например, main)
git pull --rebase           # чтобы пуш был fast-forward
gh auth status              # gh залогинен (нужно для GitHub Release)
```
Если дерево грязное — сначала закоммить/разберись (`/pause` или твой инструмент коммита).

## Шаг 2. Обнови README (все языки)

Приведи `README.md` в соответствие с реальностью: статус фаз, рабочие фичи, инструкции. Если двуязычный, держи
обе версии синхронными. Не выдумывай — отражай только то, что действительно сделано и проверено (сверься со
`STATUS.md` и закрытыми `bugs/`/`plans/ideas/` `*_DONE_*`).

## Шаг 3. Перегенерируй отрендеренные документы

`<Перегенерируй любые отрендеренные артефакты, например README.pdf (node tools/readme-pdf.mjs). Для собственного
проекта этого фреймворка также перегенерируй самораспаковывающееся ядро: node tools/build-framework.mjs.>`

## Шаг 4. Контрольная сборка (перед релизом)

`<Запусти сборку проекта (BUILD_COMMAND). Она должна пройти успешно. Это ловит ошибки ДО поднятия версии, чтобы
не оставить полу-выпущенную версию.>`

## Шаг 5. Закоммить изменения документов/сборки (перед релизом)

Закоммить обновления README/документов, чтобы коммит `release: X.Y` был чистым поднятием версии:
```bash
node tools/commit.mjs "docs: README for release X.Y"
```

## Шаг 6. Публикация (после подтверждения владельца)

`<Запусти свой релизный процесс. Если у тебя есть релизный инструмент (например, tools/release.mjs, который поднимает версию,
собирает, переименовывает артефакт, коммитит "release: X.Y", ставит тег vX.Y, пушит и запускает gh release create с
авто-нотами), запусти его. Иначе сделай это явно:>`
```bash
# подними версию (в version.json или твоём манифесте), затем:
git commit -am "release: X.Y" && git tag vX.Y && git push && git push --tags
gh release create vX.Y --title "vX.Y" --generate-notes <ARTIFACT(S) if any>
```

## Шаг 7. Проверь и отчитайся

```bash
gh release view vX.Y        # релиз существует, артефакты приложены
git log --oneline -3        # коммит релиза + тег видны
```
Отчитайся владельцу: версия, ссылка на релиз, что приложено. Готово.

## Заметки
- Релизы поднимают minor/major; обычные рабочие коммиты поднимают build/patch.
- Если пуш отклонён (non-fast-forward) — `git pull --rebase` и повтори. На шаге 6 это критично:
  тег может уже существовать локально — проверь `git tag` и `git tag -d vX.Y` перед повтором.
- НИКОГДА не делай force-push и никогда не удаляй чужие теги/релизы. Если во время публикации что-то идёт не так — остановись
  и покажи владельцу, не «чини» это вслепую.
- Не релизь в автономном режиме — только по явному запросу владельца.
