# /report-bug — завести документ бага в bugs/ (агент ведёт свой беклог багов)

Всякий раз, когда агент пишет код, тестирует его и **натыкается на баг** (краш, неверное поведение, артефакт,
регрессия, дефект библиотеки), он заводит ОТДЕЛЬНЫЙ md-документ в `bugs/` по тем же правилам, что и
существующие документы по багам. Так агент накапливает себе беклог багов — ничего не теряется, и к любому багу
можно вернуться (или передать его в `/bug-research`).

> Правило проекта (`AGENT_GUIDE.md`): «для каждого бага, даже мелкого — отрефлексируй и зафиксируй знания в
> отдельном md в `bugs/`». Этот навык как раз это и автоматизирует.

## Когда вызывать

- Агент увидел дефект во время разработки/теста, который он НЕ чинит прямо сейчас (или чинит, но хочет
  зафиксировать форензику/постмортем).
- Баг нужно отложить (взять другую задачу), не потеряв его.
- Владелец просит завести баг.
- НЕ для затыка «застрял из-за непонимания» (это `PHILOSOPHY.md`) и не вместо исправления тривиальной опечатки.

## Что делать

1. **Определи следующий номер.** `ls bugs/` → максимальный двузначный `NN` + 1. Имя файла:
   `bugs/NN_<short_english_name>.md` (snake_case, как у соседей; БЕЗ тега `DONE` — баг открыт).

2. **Собери факты ПЕРЕД написанием** (не выдумывай — документ бага ценен своими фактами):
   - Симптом: что именно наблюдается (и чем это отличается от ожидаемого).
   - Воспроизведение: детерминированные шаги. По возможности — через стенд, чтобы баг воспроизводился без человека.
   - Форензика: логи / файл краша / измерения — приложи ключевые строки (стек, сообщение abort, коды
     ошибок, размеры).
   - Сборка/версия, окружение, режим.

3. **Напиши документ по канону** (структура как у существующих документов по багам):
   ```
   # Bug NN — <однострочное описание>

   **Status:** 🔴 OPEN   (или 🟡 partial / 🔬 research-only / 🔧 fix pending verification)
   **Version/build:** <build>   ·   **When/context:** <дата, во время какой задачи найдено>

   ## Symptom
   <что наблюдается>

   ## Repro (deterministic)
   <шаги; команды стенда, если есть>

   ## Forensics
   <ключевые логи / краш / измерения>

   ## Root cause / Hypotheses
   <причина, если ясна; иначе ранжированные гипотезы — НЕ патчи вслепую>

   ## Fix plan (or the fix, if done)
   <шаги; связь с архитектурой / другими багами>

   ## Links
   <связанные баги / идеи / интервью>
   ```
   Следуй `BUG_FIXING_FRAMEWORK.md`. Если баг в сторонней библиотеке, заведи тикет им (например,
   через `gh`) и сошлись на него из документа.

4. **Зафиксируй в беклоге/процессе:**
   - Если важный/блокирующий — короткая строка в `STATUS.md`.
   - Закоммить (в автоциклах — по обычной дисциплине): `node tools/commit.mjs "docs(bugNN): …"`.

5. **Жизненный цикл:** пока открыт — заводи БЕЗ `DONE`. Когда ПОДТВЕРЖДЁННО закрыт (исправлен и проверен) — переименуй
   `git mv bugs/NN_x.md bugs/NN_DONE_x.md` и допиши раздел `## ✅ STATUS: DONE (date)` (что
   сделано / как проверено). Ревизия беклога — навык `/check-backlog`.

## Заметки
- Лучше завести баг и оставить его открытым, чем потерять. Фактическая точность важнее красивого текста.
- Если баг сопротивляется (≥3 слепых попыток исправления) — переключись на `/bug-research` (исследование без кода) по тому же документу.
