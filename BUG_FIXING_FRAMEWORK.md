# BUG_FIXING_FRAMEWORK — как агент чинит дефекты

Чтобы починить баг, агент должен:

- **Сосредоточиться только на этом одном баге.** Не переделывай весь мир; не чини попутно три других вещи.
- **Вести повествование в чате**, хотя бы немного, на естественном языке — что ты делаешь прямо сейчас —
  чтобы человек мог глянуть и понять, где сейчас работа.
- **Рефлексировать и фиксировать знание** по каждому багу, даже мелкому, в отдельном markdown-файле на
  каждый баг в каталоге `bugs/`.
- **Intent gate перед первой правкой поведения** (fable-method, шаг 4): напиши одну строку —
  `INTENT: code does <X>; the failing check/task expects <Y>; the spec (README/docs/docstring) says <Z>`
  — реально открыв спеку для третьего слота. Если X/Y/Z расходятся, это расхождение И ЕСТЬ находка:
  «баг» может жить в проверке или в формулировке задачи, а не в коде. Никогда не подгоняй молча одну
  сторону под другую; порядок авторитетов: явное слово владельца > спека > тесты > текущее поведение кода.
- **Войти в цикл:** запусти приложение → воспроизведи баг → прочитай логи по этому багу → выдвини догадку о
  причине → внеси *одно, точечное* изменение → собери → снова запусти приложение → снова попробуй воспроизвести.
- Суть: **точечные изменения, затем сборка, чтобы проверить, помогло изменение или нет.**
- Упрощённо: Фикс → Тест → Чтение логов → Фикс → Тест → Чтение логов → … пока не заработает правильно.
  Работает правильно **и есть критерий приёмки** — в этот момент баг считается починенным. «Работает
  правильно» — это *наблюдение* (запустилось, отрисовалось, посчиталось), а не вывод из чтения диффа:
  непроверенное «починено» — классический фрод, ради которого существует `/fable-judge`.
- **Twin check после фикса** (fable-method, шаг 5c): дефект, найденный в одном месте, предположительно
  повторяется в других, пока ты не поискал. Назови точную ошибочную конструкцию, обыщи весь проект и
  запиши в документ бага (и в отчёт): `TWINS: searched <паттерн> — found <N> other sites: <файлы или
  "none">`. Почини их или перечисли — заявление о полноте без поиска за ним пусто.
- Чтобы починить баг, часто полезно **поискать в вебе** решение — форумы, GitHub issues, Reddit, Stack
  Overflow, официальная документация.

> ⚠️ **ПРАВИЛО 3 ПОПЫТОК → переход к исследованию (`/bug-research`).**
> Если после **трёх итераций** «точечный фикс → сборка → тест» баг НЕ починен — СТОП тыкать вслепую и
> наугад. Дальнейшие случайные попытки тратят время и сборки, и (если тест зависит от внешних сервисов)
> ненадёжны. Вместо этого запусти навык **`/bug-research`**: глубокий поиск в вебе с сырой базой знаний,
> записанной в документ бага, чтение и анализ кода БЕЗ правок, чтобы локализовать причину, рефлексия и
> обоснованная гипотеза + план. Возвращайся к починке только тогда, когда поймёшь причину. Навык:
> `.claude/skills/bug-research/SKILL.md`.
>
> 🧠 **И, ЧТО ВАЖНЕЕ ВСЕГО (см. `PHILOSOPHY.md`):** долгий затык почти всегда означает, что ты делаешь
> ФИКС СЛИШКОМ СЛОЖНЫМ, потому что НЕ ПОНЯЛ задачу — а НЕ что задача сложна или библиотека «глючная».
> Библиотеки просты; всё уже придумано до нас. Если фикс громоздкий, с кучей флагов и обходных путей — это
> тревожный сигнал: остановись и ПЕРЕОСМЫСЛИ задачу простыми словами, найди простой поддерживаемый путь
> (KISS + Оккам). Затык = сигнал упростить понимание, а не «копать упорнее».

- Для эффективной починки код должен быть **прокомментирован**.
- Для эффективной починки ты ОБЯЗАН рефлексировать и записывать своё знание о работе над этим багом в
  отдельный документ именно про *эту конкретную* работу по починке. Такие размышления следует и должно
  хранить отдельными markdown-документами в каталоге `bugs/`.
- Как только баг понят или починен (и всегда после стопа по правилу 3 попыток) — **зафиксируй
  переиспользуемый урок** в `EXPERIENCE.md` (навык `/experience`): вывод уровня подхода («X провалился
  из-за R; сработал Y»), а не деталь дефекта (она остаётся в документе `bugs/`). Так следующая сессия не
  пойдёт тем же тупиком.
- Чтобы иметь возможность взаимодействовать с тестируемой программой, ты можешь и должен **устанавливать
  или собирать дополнительные инструменты**, которые позволяют агенту наблюдать за софтом и управлять им:
  видеть его вывод, инспектировать его состояние, детерминированно воспроизводить дефект и прогонять его
  без человека. Вкладывайся в такую инструментовку — она окупится на каждом будущем баге.

## Инструментовка — построй тестовый стенд (харнесс), не гадай

Самый большой множитель силы для автономной отладки — это **харнесс**: инструментовка, которая позволяет
агенту воспроизводить и наблюдать баг самостоятельно, без человека в цикле и без ненадёжной ручной
инспекции.

Принципы:

- **Никогда не гадай, что делает программа — наблюдай за ней.** Добавь структурированное логирование,
  канал отладочных команд, детерминированный способ загнать софт ровно в то состояние, что вызывает баг.
- **Сделай воспроизведение детерминированным.** Баг, который можно воспроизвести по требованию, — это баг,
  который можно починить. Баг, который ты видишь «иногда», — сначала задача на исследование (`/bug-research`).
- **Предпочитай объективную проверку прикидке на глаз.** Если визуальная/ручная проверка ненадёжна
  (тонкие искажения, тайминги), придумай объективную проверку (контрольный вход известной формы, измеримый
  выход, лог размера/контрольной суммы) и запиши её в документ бага.
- **Развивай харнесс со временем.** Каждый раз, когда делаешь что-то вручную, чтобы воспроизвести или
  проверить, спроси: «могу ли я добавить команду/флаг/скрипт, чтобы в следующий раз это был один шаг?»
  Затем добавь это и задокументируй в `AGENT_GUIDE.md`. Харнесс — живой инструмент: расширяй и
  документируй его.

## Работа с логами

- Всегда сверяй **текущее время** и **временную метку файла/записей лога** перед чтением — чтобы читать
  свежие, релевантные логи, а не вчерашние.
- Фильтруй логи по багу: grep по нужной подсистеме, тексту ошибки, маркерам падения.
- Прикрепляй ключевые строки (стек-трейс, сообщение об abort, коды ошибок, размеры) в документ бага — это
  криминалистика, на которую будут опираться будущие сессии.

## Документ бага (один на баг, в `bugs/`)

Фиксируй, не пересказывай. Хороший документ бага позволяет будущей сессии (с пустым контекстом) подхватить
баг с нуля. Каноническая структура (полный шаблон — см. `/report-bug`):

```
# Bug NN — <one-line description>

**Status:** 🔴 OPEN  (or 🟡 partial / 🔬 research-only / 🔧 fix pending verification / ✅ DONE)
**Version/build:** <...>   ·   **When/context:** <date, during which task it was found>

## Symptom
<what is observed, and how it differs from expected>

## Repro (deterministic)
<steps; harness commands if available>

## Forensics
<key logs / crash / measurements>

## Root cause / Hypotheses
<the cause if known; otherwise ranked hypotheses — do NOT patch blindly>

## Fix plan (or the fix, if done)
<steps; relation to architecture / other bugs>

## Links
<related bugs / ideas / interviews>
```

Когда баг подтверждённо починен и проверен, помечай его DONE по соглашению о теге `DONE` (переименуй
`bugs/NN_x.md` → `bugs/NN_DONE_x.md` и добавь секцию `## ✅ STATUS: DONE (date)`). См.
`AGENT_GUIDE.md` → «Беклог и тег DONE» и навык `/check-backlog`.

---

## Если баг в чужой библиотеке

Если ты находишь настоящий дефект в стороннем зависимом пакете, заведи issue/тикет в их трекере (например,
через `gh` CLI, от имени человека, если есть полномочия) и сошлись на этот тикет из своего документа бага.
Это и помогает экосистеме, и документирует, почему ты сделал обходное решение.
