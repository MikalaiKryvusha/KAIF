# EXPERIENCE (ОПЫТ) — накопленный опыт ИИ-агента

> Растущий журнал уроков — **вынесенная память о том, *что работает, а что нет***, чтобы свежая сессия
> (или автономный цикл) без контекста не наступала повторно на те же грабли. **Свернись** сюда ПЕРЕД
> задачей, **допиши** ПОСЛЕ значимой попытки (успех **или** провал). Ищи грепом, не листай.
>
> **Теги стоят инлайн на каждой записи** (не в общем списке) — чтобы один греп находил опыты напрямую:
> `grep '#loop' EXPERIENCE.md` · `grep -i '#context\|#build' EXPERIENCE.md` · `grep '❌' -A4 EXPERIENCE.md`
> · `grep 'EXP-0002' EXPERIENCE.md`. Переиспользуй существующий тег, если подходит (грепни, чтобы увидеть).
>
> **Формат записи (коротко и греп-дружелюбно).** Новые — сверху. id `EXP-NNNN` · ISO-дата · маркер
> `✅`/`❌`/`❌→✅` · инлайн-`#теги`. Навык: `/experience`. Живой справочник — тегом `DONE` не помечается.

## Записи

### EXP-0007 · 2026-07-16 · ❌→✅ · #verification #windows #installer
**Ситуация:** Фаза 2 эпика 1.5 — машинерия установки «выглядела готовой» после позитивного прогона.
**Что делали:** гоняли по fable-циклу ВСЕ сценарии наблюдением: негатив verify-final, autofill на
реальном package.json, anonymous end-to-end с греп-сканом, tamper sha256, пост-очистку.
**Итог:** ✅ наблюдение вскрыло 3 реальных бага, невидимых в happy-path: (1) PowerShell пишет UTF-8 c BOM
→ JSON.parse падал → инсталлятор МОЛЧА ПЕРЕСОЗДАВАЛ package.json юзера (destroy-without-looking!);
(2) `check` парсил бандл, который verify-final удаляет — пост-проверки были бы мертвы; (3) анонимная
установка утекала автора через навык kaif-version, заметку в шаблоне и origin-URL в ядре.
**Урок:** «работает» = наблюдение КАЖДОГО заявленного свойства, включая негативные пути и жизнь ПОСЛЕ
самоочистки. На Windows любой JSON-парс входных файлов — только BOM-терпимым ридером; файлы юзера при
нечитаемости пропускаем с предупреждением, НИКОГДА не перезаписываем. Анонимность держится только
механическим греп-гейтом в финале, не дизайном «вроде всё учли».   → plans/13 Фаза 2, EXP-0005

### EXP-0006 · 2026-07-16 · ✅ · #integration #vendoring #attribution
**Ситуация:** встраивание чужого MIT-фреймворка (fable-method) в KAIF (эпик 1.5, Фаза 1).
**Что делали:** гибрид: 4 навыка вендорены ДОСЛОВНО с шапкой-атрибуцией (автор/репо/версия-пин/MIT) и
минимальными ПОМЕЧЕННЫМИ патчами (domains→сферы KAIF, GSD→циклы KAIF, suite→апстрим); в свои документы —
только точки вызова; конфликты (коммит-политика, интервью-vs-вопрос, циклы-vs-loop) сняты явной сшивкой.
**Итог:** ✅ сборка зелёная (43 блока), формулировки апстрима не тронуты.
**Урок:** чужие выверенные тексты НЕ парафразить — их сила в отточенных формулировках (урок v1→v3 самого
fable: проза 1/4 → принудительный артефакт 4/4). Вендорь дословно + шапка-атрибуция со СПИСКОМ патчей +
пин версии; расхождения с апстримом должны быть перечислимы. Конфликты двух канонов снимай явной строкой
сшивки, а не молчаливым выбором.   → researches/03 §6, интервью 002 Q1/Q2

### EXP-0005 · 2026-07-16 · ❌→✅ · #build #windows #crossplatform
**Ситуация:** первая пересборка ядра на Windows-машине: `build-framework.mjs` упал с ENOENT '...'.
**Что делали:** нашли, что регэксп срезки авторского комментария требует `-->\n`, а CRLF-чекаут даёт
`-->\r\n`; пример `{{EMBED:...}}` из несрезанного комментария уходил в обработку маркеров.
**Итог:** ✅ оба регэкспа → `\r?\n`; сборка зелёная (bugs/03).
**Урок:** инструменты репозитория обязаны быть терпимы к CRLF: в регэкспах по концам строк всегда
`\r?\n`, не голый `\n` (рабочая машина владельца теперь Windows, autocrlf=true — дефолт). Симптом
«инструмент читает буквальный плейсхолдер» = ищи несработавшую срезку/экранирование выше по конвейеру.
   → bugs/03_DONE_build_breaks_on_crlf_checkout.md

`gh release view <prev> -q .body` и повторяй форму.   → framework/skills/release/SKILL.md
