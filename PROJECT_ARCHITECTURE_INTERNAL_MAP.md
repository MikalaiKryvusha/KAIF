# Krinik AI Framework (KAIF) — внутренняя карта архитектуры

Логическая архитектура *этого* репозитория: абстракции, которыми оперирует проект (сфера — «мета/фреймворк»),
их суть и взаимодействия. Спутник — `PROJECT_STRUCTURE_EXTERNAL_MAP.md` (внешняя структура: где что лежит).

---

## Ключевые абстракции

| Абстракция | Что это (суть) | Ответственность |
|------------|----------------|-----------------|
| **Полезная нагрузка** (`framework/`) | Канонические универсальные шаблоны (EN) | Единственный источник истины того, что разворачивается в чужие проекты. |
| **Ядро** (`KAIF.md`) | Самораспаковывающийся документ | Сгенерированный из полезной нагрузки инсталлятор + описание; отдаётся пользователю. |
| **Повествовательный стержень** (`framework/_intro.md`) | Проза ядра с маркерами `{{…}}` | Задаёт структуру и инструкции распаковки; в него встраиваются шаблоны. |
| **Встраиваемый блок** (`> **FILE: …**`) | Помеченный файл-шаблон в ядре | Говорит распаковывающему агенту, какой файл создать в проекте. |
| **Навык** (`framework/skills/<name>/SKILL.md`) | Повторяемый ритуал с YAML-фронтматтером | Глагол работы над проектом (`/resume`, `/pause`, циклы, `/revision`, …). |
| **Сфера** (`framework/spheres/*`) | Библиотека терминов домена | Адаптирует терминологию развёртывания к сфере (программирование, наука, …). |
| **Адаптер** (`framework/adapters/*`) | Проводка под агентскую систему | Куда класть контекст и как транслировать навыки (Claude Code, Codex, …). |
| **Обвязка** (корневые доки + `.claude/`) | KAIF, применённый к самому себе | Dogfooding: ведёт разработку самого фреймворка. |
| **Маркер** (`.kaif/kaif.json`) | Запись развёртывания | Версия, дата, origin, tracking, сфера, агент. |

## Как они взаимодействуют (поток генерации)

```
framework/_intro.md ─┐
framework/<DOC>.md   │   tools/build-framework.mjs
framework/readmes/*  ├───────────────────────────▶  KAIF.md  ──(валидатор)──▶  tools/check-framework.mjs
framework/skills/**  │   (подставляет {{VERSION}},        (сбалансированные
                     ┘    встраивает FILE-блоки)           FILE-блоки + ограждения)
```

- `build-framework.mjs` оборачивает каждый шаблон в ограждение из 6 обратных кавычек (чтобы 3-кавычные
  блоки внутри шаблонов не «протекали») и помечает `> **FILE: \`<dest>\`**`.
- `check-framework.mjs` проверяет: число FILE-блоков = (ключевые доки + README директорий + навыки),
  ограждения сбалансированы, не осталось маркеров `{{…}}`, каждый навык встроен.
- Версия берётся из `version.json` как `major.minor` (две цифры; дата — в описании релиза, не в имени).

## Инварианты модели

- **Полезная нагрузка — единственный источник.** `KAIF.md` **генерируется**; править руками нельзя.
- **Два слоя не смешиваются.** Улучшение *шаблона* → правь `framework/` и пересобирай; улучшение *ведения
  этого проекта* → правь корневую обвязку. Обычно двигать оба синхронно.
- **Язык.** Полезная нагрузка — английская; обвязка — русская; содержание держим синхронным.
- **DRY.** Шаблон живёт один раз в `framework/` и встраивается в ядро, не дублируется руками.

## Ключевые решения, зашитые в архитектуру

- Само-извлечение через один файл: всё нужное для развёртывания — внутри `KAIF.md` (см. `MASTER_PLAN.md`,
  лог решений).
- Этапная распаковка (v1.1): структура → изучение проекта (`KAIF_DEPLOYMENT_PLAN.md`) → адаптация — чтобы
  слабые ИИ-системы не захлёбывались контекстом.
