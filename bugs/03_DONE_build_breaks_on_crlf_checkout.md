# Bug 03 — сборка падает на Windows-клоне (CRLF): ENOENT '...' из {{EMBED:...}}

**Status:** ✅ DONE
**Version/build:** v1.4 (build на момент фикса) · **When/context:** 2026-07-16, первая пересборка ядра
на новой локальной Windows-машине владельца (Фаза 1 эпика 1.5).

## Symptom

`node tools/build-framework.mjs` падает: `Error: ENOENT: no such file or directory, open
'D:\work\ai_sandbox\KAIF\...'` — сборщик пытается открыть буквальный путь `...`.

## Repro (deterministic)

Клонировать репозиторий на Windows с `core.autocrlf=true` (дефолт установщика Git) → рабочая копия
получает CRLF-переводы строк → `node tools/build-framework.mjs`.

## Forensics

`embedFile` получает relPath `...` из маркера `{{EMBED:...}}` в **строке 4 `_intro.md`** — это ПРИМЕР
синтаксиса внутри авторского комментария `<!-- ... -->`, который сборка обязана срезать ДО обработки
маркеров. Регэксп срезки — `/^<!--[\s\S]*?-->\n/` — требует, чтобы за `-->` шёл `\n`; при CRLF там
`\r\n`, матч не происходит, комментарий остаётся, и пример-маркер уходит в обработку.

## Root cause

Инструмент предполагал LF-переводы строк (репозиторий жил на macOS). На Windows-чекауте с
`autocrlf=true` предположение ломается. Блобы в git при этом хранятся LF — ломается только working tree.

## Fix

`tools/build-framework.mjs`: оба регэкспа срезки комментария (`_intro.md` и `KAIF-SLIM.md`) →
`/^<!--[\s\S]*?-->\r?\n/` + комментарий-пояснение. Проверено: сборка зелёная, 43 блока, fences
сбалансированы, лишних маркеров нет.
