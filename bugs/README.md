# bugs/ — собственный беклог дефектов и процессов агента

По одному markdown-документу на баг или дефект процесса, согласно канону из `BUG_FIXING_FRAMEWORK.md` и
навыка `/report-bug`. Смысл: каждый дефект — даже маленький — становится долговечным артефактом (симптом →
детерминированное воспроизведение → форензика → корневая причина/гипотезы → план исправления → ссылки), чтобы ничего не терялось и
будущая сессия (с пустым контекстом) могла подхватить его с нуля.

## Соглашения

- Имя файла: `NN_<short_english_name>.md` (число с ведущими нулями, snake_case), например `03_fence_imbalance.md`.
- **Открыт** = в имени нет тега `DONE`. **Закрыт и проверен** = `git mv NN_x.md NN_DONE_x.md` и добавить
  раздел `## ✅ STATUS: DONE (date)` (что сделано / как проверено). См. `AGENT_GUIDE.md` → «Беклог и
  тег DONE» и навык `/check-backlog`.
- Этот `README.md` — справочный документ, а не задача — никогда не помечайте его тегом DONE.

На v1.0 нет открытых багов. Этот каталог — часть dogfooding фреймворка: проект фреймворка
ведёт собственный беклог багов ровно так, как он велит это делать развёрнутым проектам.
