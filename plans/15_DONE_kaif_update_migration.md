# План 15 — Механика уважительного обновления KAIF (идея 14)

> Источник: `ideas/14_how_to_update_kaif_to_1.5.md` (принята в работу 2026-07-16). Дата: 2026-07-16.
> Контекст: у владельца несколько проектов обвязаны KAIF 1.4; «ручное» когнитивное обновление
> развёрнутого KAIF агентом — колоссально трудная работа. Владелец подсказывает направление:
> модульность + умная mjs-машинерия. Архитектура Thin KAIF (план 13, Фаза 2) строилась с этим прицелом:
> `.kaif/kaif-core.mjs` живёт в проекте постоянно и знает, что развёрнуто.

## Цель

Обновление развёрнутого KAIF до свежей версии должно быть **преимущественно механическим**: машинерия
сама решает, что можно безопасно заменить, что нельзя трогать, и оставляет агенту короткое когнитивное
задание только по реально разошедшимся местам. Плюс — рабочий путь для **legacy-проектов (1.4 и
старше)**, где машинерии ещё нет.

## Дизайн (KISS)

### 1. Модель принадлежности файлов — три класса

| Класс | Примеры | Политика обновления |
|---|---|---|
| **Контент владельца** | `GOAL.md`, `STATUS.md`, `MASTER_PLAN.md`, карты, `EXPERIENCE.md`, все `plans/ ideas/ bugs/ researches/ interviews/ homeworks/` (кроме README) | **НИКОГДА не трогается.** |
| **Framework-owned, нетронутый с установки** | навыки, `PHILOSOPHY`, `BUG_FIXING`, `TESTING_FRAMEWORK`, README директорий, сферы, references | Перезаписывается новой версией **механически** (с автозаполнением плейсхолдеров, как при install). |
| **Framework-owned, но разошедшийся** (агент/владелец правили после установки) | чаще всего `AGENT_GUIDE.md`, кастомизированные навыки | НЕ перезаписывается; попадает в `KAIF_UPDATE_TASK.md` — когнитивная сшивка агентом. |

**Механика различения** — sha256-слепок содержимого «как записано при install/update» сохраняется в
`.kaif/deploy-manifest.json` (`shas: {path: sha}`). При update: sha диска == слепок → файл никто не
трогал → безопасная механическая замена; иначе — разошёлся → в задание.

### 2. Команда `update` в `KAIF-CORE.mjs`

```
node .kaif/kaif-core.mjs update [--channel release|main] [--source <dir|url>]
```

1. Скачивает свежие `kaif-manifest.json` + `KAIF-CORE.mjs` + бандл (sha256-верификация — как в
   загрузчике; ~30 строк переиспользованной логики: ядро не может зависеть от удалённого при
   verify-final загрузчика).
2. Сравнивает версии (`.kaif/kaif.json` vs манифест); равные → «уже свежий», выход.
3. Классифицирует по модели §1 и применяет: новые файлы — добавить; нетронутые — заменить (новый
   шаблон + автозаполнение); разошедшиеся — пропустить и записать в задание; контент владельца — не
   трогать по определению (его нет в манифесте framework-owned путей).
4. Заменяет **само ядро** (`.kaif/kaif-core.mjs`) новой версией, обновляет `deploy-manifest` (пути+sha)
   и `kaif.json` (версия/дата).
5. Эмитит **`KAIF_UPDATE_TASK.md`** с чекпоинтами `KAIF-UPDATE:` (по образцу задания адаптации):
   список разошедшихся файлов («сшей новые изменения шаблона со своими правками — вот что изменилось»),
   новые сущности, требующие внимания (например, новые навыки → упомянуть в своих доках), прогон
   `node .kaif/kaif-core.mjs check` и judge-проход. `verify-final`-аналог: `update-verify` (чекпоинты →
   очистка временного).
6. Anonymous-проекты: у них ядра нет (удалено by design) → путь через повторный бутстрап (см. §3),
   режим сохраняется из `kaif.json` (`tracking: anonymous`).

### 3. Legacy-путь: 1.4 (и старше) → 1.5+

В 1.4-проекте нет ни ядра, ни deploy-manifest. Путь: владелец кладёт **тонкий `KAIF.md`** и говорит
«обнови KAIF» → бутстрап (3 шага `KAIF-BOOT:`) → загрузчик → ядро `install` **детектит существующий
`.kaif/kaif.json` старой версии** и переключается в режим обновления:

- `writeIfNew` уже сейчас не трогает существующие непустые файлы → контент владельца в безопасности
  by design; новые файлы 1.5 (fable-навыки, references, сферы, `TESTING_FRAMEWORK.md`) добавляются;
- вместо `KAIF_ADAPTATION_TASK.md` эмитится `KAIF_UPDATE_TASK.md`: без слепков различить «нетронутый
  шаблон 1.4» и «правленый» нельзя → все существующие framework-owned файлы перечисляются как
  потенциально разошедшиеся, агенту даётся список «что нового в шаблонах 1.5» (краткий changelog
  шаблонов в бандл-манифесте `templateNotes`) для точечной сшивки;
- по завершении — обычный `verify-final`, после которого проект получает ядро + слепки, и все
  **последующие** обновления становятся механическими по §2.

### 4. Обновление навыка `/kaif-update`

Шаблон навыка переписывается с «когнитивной миграции» на: «(а) есть `.kaif/kaif-core.mjs` → запусти
`update`; (б) нет (legacy/anonymous) → положи свежий тонкий `KAIF.md`, пройди бутстрап; (в) отработай
`KAIF_UPDATE_TASK.md` с чекпоинтами». `kaif:update`-хендл → `node .kaif/kaif-core.mjs update`.
