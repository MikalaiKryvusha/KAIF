> ответы Криника в `plans/ideas/01`, `02`.
> **Статус:** живой план (обновляется по ходу). **Дата:** 2026-06-30.
> Это блюпринт превращения v1.0-дистилляции (бренд «KAIF») в **KAIF** с полным жизненным циклом.

---

## 0. Принципы реализации

- **KISS + Оккам** (см. `PHILOSOPHY.md`): каждая «ручка» — минимально необходимая, без лишних сущностей.
- **Уважительность** — ключевое слово всего жизненного цикла: обновление, форк, переключение и удаление
  KAIF никогда не ломают проект пользователя; его файлы не трогаем, только аккуратно работаем с обвязкой KAIF.
- **Самораспаковка сохраняется**: всё необходимое для развёртывания — внутри `FRAMEWORK.md` (генерируется).
- **Два слоя** (полезная нагрузка `framework/` → `FRAMEWORK.md` vs. обвязка проекта) — не путать.
- **Коммитим часто**, оставляем репозиторий в рабочем состоянии после каждого этапа.

## 2. Версионирование и жизненный цикл (Идея 04) — Фаза L

**Маркер развёрнутого KAIF.** При инжекции в проект создаётся `.kaif/kaif.json`:
```json
{ "framework": "KAIF", "version": "X.Y.Z", "released": "YYYY-MM-DD",
  "origin": "https://github.com/MikalaiKryvusha/KAIF", "tracking": "origin", "sphere": "<sphere>", "agent": "<agent>" }
```
Так ИИ-агент в проекте всегда знает свою версию, origin и режим (origin/fork).

**Семантическое версионирование:** `MAJOR.MINOR.PATCH` + дата релиза. Простое правило в `AGENT_GUIDE`/доках.

**Навыки жизненного цикла** (новые `framework/skills/` + обёрточные `.claude/skills/`):
- `kaif-version` — показать текущую версию KAIF в проекте; сверить с origin (последний релиз/тег на GitHub);
  если есть свежее — сообщить.
- `kaif-update` — **уважительное миграционное обновление** из origin: показать диффы версий, предложить
  человеку («Вижу новую версию KAIF — провести уважительное обновление и миграцию?»), при согласии —
  мигрировать ядро, сохранив локальные кастомизации и все контентные артефакты.
- `kaif-fork` — сделать «слепок» текущего KAIF проекта в **свой** GitHub-репозиторий и переключить
  `tracking` на `fork` (дальше версионирование ведётся «под себя»).
- `kaif-switch-origin` — вернуть `tracking` на официальный origin с уважительной миграцией.
- `kaif-remove` — **уважительное удаление** ядра KAIF (инструкции, инструменты, навыки, обвязка), но
  **сохранить контентные артефакты**: `bugs/`, `interviews/`, `plans/` (homework, ideas).
- `kaif-remove-all` — **полное** уважительное удаление: ядро + артефакты. Проект пользователя остаётся
  целым и рабочим; KAIF «выжигается» из истории, словно его не было.

**npm-run «ручки».** При инжекции KAIF уважительно расширяет (или создаёт) `package.json` проекта,
добавляя скрипты-обёртки над `kaif/` инструментами, не ломая существующие:
`kaif:version`, `kaif:check`, `kaif:update`, `kaif:fork`, `kaif:switch-origin`, `kaif:remove`, `kaif:remove-all`.
При удалении KAIF эти скрипты вычищаются. (Реализация: merge в `scripts`, маркировка блока KAIF.)

## 3. Мультисферность и библиотеки терминов (Идея 04) — Фаза S

- KAIF применим не только к программированию: математика, физика, космос, биология, медицина, социология,
  образование, дизайн, проджект-менеджмент, реклама, спорт, питание, финансы, право, музыка, писательство и т.д.
- **Определение сферы при деплое:** агент изучает проект и/или спрашивает человека о сфере; пишет её в
  `.kaif/kaif.json` → `sphere`.
- **Сфера-специфичная адаптация:** разворачиваемые доки/навыки/каталоги адаптируются под сферу
  (терминология, что считать «багом»/«релизом»/«тестом» в этой сфере).
- **Библиотеки терминов:** `framework/spheres/<sphere>.md` — краткое тезисное введение в сферу + словарь
  ключевых терминов + маппинг базовых сущностей KAIF на язык сферы (например, `bugs/` → «дефекты/наблюдения/
  отклонения»). Плюс `framework/spheres/_index.md` и `framework/spheres/_template.md`. На v1.0 заводим
  несколько эталонных сфер (programming + 3–5 примеров) и шаблон для остальных.

## 4. Мультиагентность: слой трансляции/адаптации (Идея 02) — Фаза A

- Содержимое навыков **переносимо**; специфична только «обвязка обнаружения» (где агент берёт контекст и
  команды). Делаем `framework/adapters/<agent>.md` — как развернуть KAIF под конкретную систему.
- Приоритет: **OpenAI Codex, GitHub Copilot, Cursor, Windsurf, Cline, Roo Code**; по остаточному принципу —
  **Devin, OpenCode, Aider, Junie**.
- Для каждой системы: куда кладётся контекст-файл (аналог `CLAUDE.md`/`AGENTS.md`), как оформляются команды
  (аналог `.claude/skills/`), особенности. Универсальный фолбэк — `AGENTS.md`, указывающий на `AGENT_GUIDE.md`.
- При деплое: агент определяет/спрашивает целевую агентскую систему → транслирует зависимые модули в её формат.
- `framework/adapters/_index.md` — обзор + матрица соответствий механизмов Claude ↔ другой системы.

## 5. Валидатор (Идея 01) — Фаза V

- `tools/check-framework.mjs` — проверяет `FRAMEWORK.md`: число встроенных `FILE:`-блоков = (число доков +
  число навыков), сбалансированность 6-кавычных ограждений, отсутствие незакрытых `{{…}}` и `<PLACEHOLDER>`,
  существование всех встроенных файлов в `framework/`. Ненулевой код при ошибке.
- `npm test` — атомарный запуск валидатора (независимая сущность, запускается когда угодно).
- Встроить вызов в конец `tools/build-framework.mjs` — самопроверка сборки/распаковки.
