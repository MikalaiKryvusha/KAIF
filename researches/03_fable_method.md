## 1. Что это (факты)

**The Fable Method** — дистилляция *способа работы модели Claude Fable 5*, записанная как набор навыков,
которые может исполнять любая (в т.ч. слабая) модель. Автор — Sahir619, сообщество (НЕ Anthropic).
Лицензия **MIT**. Форма — **плагин Claude Code** (и собственный marketplace) + переносимый `AGENTS.md`.

| Факт | Значение |
|---|---|
| Репозиторий | github.com/Sahir619/fable-method |
| Версия | v1.4.0 (2026-07-15) — **4 релиза за 10 дней** (1.0.0 от 2026-07-06) |
| Активность | Высокая; первый внешний контрибьютор уже влит (artifact gate, PR #2) |
| Лицензия | MIT — совместима с KAIF (MIT), требует сохранения копирайта при копировании |
| Дисциплина проекта | «Ни одно правило не входит без предварительно провалившегося теста» (CONTRIBUTING) |
| Доказательная база | 15 раундов eval, 260+ прогонов агентов, 14 фикстур-ловушек, сырые вердикты судей в `eval/results/` |

Главный тезис: **средняя модель, следующая этому циклу, бьёт сильную модель во фристайле** — качество
живёт в структуре, свидетельствах и честности, а не в модели. Ценность концентрируется на «ловушках»
(конфликт авторитетов, ложные заявления о завершении, слабые исполнители, безнадзорные прогоны); на
обычных задачах с сильной моделью прироста нет — и **нули опубликованы честно** рядом с победами.

## 2. Устройство репозитория

Четыре навыка, одна философия: **think / act / prove / grow**.

| Навык | Роль | Размер | Суть |
|---|---|---|---|
| `fable-method` | **think** | ~18K зн. + references | 7-шаговый цикл решения задачи + гейты (см. §3) |
| `fable-loop` | **act** | ~5.6K | Оркестровка цикла: параллельные evidence-субагенты → один план → хирургическое исполнение в main thread → **адверсарные верификаторы** → аудированный отчёт |
| `fable-judge` | **prove** | ~6.1K | Адверсарная проверка «готовой» работы: отчёт = набор заявлений, не доказательство; перепрогон всех проверок, diff как ground truth, охота на 6 классических фродов; вердикты VERIFIED / CAVEATS / REFUTED. Режим `suite` — прогон ловушек по любому навыку/модели (требует `eval/`) |
| `fable-domain` | **grow** | ~10.6K | Генератор доменных адаптеров-бандлов: обсуждение → исследование с источниками → адаптер + ловушка + smoke-eval; жёсткие red-lines (медицина, юр./фин. советы — отказ) |

Обвязка:
- `references/` у fable-method: `failure-modes.md` (18 типовых провалов агентов → какой шаг лечит),
  `examples.md` (разобранные примеры по формам задач), `flowcharts.md` (8 mermaid-схем — «исполняемый
  псевдокод»), `domains/` (**8 доменных адаптеров**: marketing, research, data analysis, business-ops,
  finance, legal-compliance, design-ux, devops + `TEMPLATE.md` — их схема; кодинг — домен по умолчанию;
  медицина исключена намеренно). Адаптер меняет «существительные», не цикл: что считается
  свидетельством, кто авторитет, что значит «проверить наблюдением», таблица фродов сферы, и
  **обязательный минимум свидетельств** (binding minimum evidence set).
- `AGENTS.md` (~16.6K) — тот же метод **без Claude-специфики**, для любого харнесса (Codex, Cursor,
  aider, сырой системный промпт): «вставь в инструкции агента или положи в корень как AGENTS.md».
- `eval/` — харнесс честности: 14 сценариев-ловушек с GROUND-TRUTH (агенту под тестом не выдаются),
  workflow.js, журнал RESULTS.md, сырые json-вердикты. Объёмный (сотни КБ) — это НЕ полезная нагрузка
  для пользователя метода, это доказательная база репозитория.
- Установка: (а) **плагин** `/plugin marketplace add Sahir619/fable-method` + `/plugin install
  fable@fable-method` — все 4 навыка, версионируемо, обновляемо; (б) **standalone** `install.sh/.ps1` —
  копирует **только 3** навыка (без fable-domain) в `~/.claude/skills/`; (в) **другие агенты** —
  `AGENTS.md`. Рекомендация автора: сделать метод проактивным строчкой в глобальном `CLAUDE.md`.

## 3. Сам метод (ядро, что именно «встраивать»)

Цикл: `0 классифицируй запрос → 1 определи "готово" → 2 собери свидетельства → 3 реши и зафиксируйся
→ 4 действуй хирургически → 5 проверь наблюдением → 6 доложи результатом-вперёд`. Перед циклом два
гейта: **triviality gate** (тривиальное — сделай, проверь одной очевидной проверкой, доложи в 2
предложениях) и **fit gate** (маршрутизация «где живёт ответ»: источники → цикл; неизвестная техника →
сначала исследуй; только собственная догадка → честно скажи, не наряжай догадку в строгий процесс;
специализированная повторяющаяся процедура → сделай навык).

Ключевая инженерная находка — **принудительные артефакты на точках решения** (forced artifacts):
- `INTENT: code does X; check expects Y; spec says Z` — перед любой сменой поведения (обязан вскрыть
  спеку; если X/Y/Z расходятся — это и есть главная находка, не правь молча);
- `AUTH: user said "<точная цитата>"` — перед необратимым/внешним действием; **документация — не
  авторизация**, установленный навык — не авторизация, «завершить задачу» — не авторизация;
- `TWINS: searched <паттерн> — found N other sites` — после фикса дефекта обыщи проект на близнецов;
- `PENDING: <действие> — awaiting your authorization` — предписанный доками, но сознательно не
  сделанный шаг;
- **artifact gate** — финальная механическая промётка отчёта: что «должен» и чего нет — добавь.
Плюс порядок авторитетов (слово юзера > спека > тесты > текущее поведение кода), лестница восстановления
после неудачной правки, жёсткие границы (3 проваленных цикла fix-verify → стоп и возврат юзеру; 2
пустых поиска → хватит искать), запреты по умолчанию (не коммить/пушить без указания, не ослаблять
проверки, не трогать секреты, не добавлять зависимости, не удалять вне заявленного скоупа).

## 4. Чем доказано (и границы доказанности)

| Результат | Цифры | Раунд |
|---|---|---|
| Слабая модель (Haiku) начинает вскрывать конфликт «спека vs тест» вместо молчаливой «починки» правильного кода | 0/4 → **4/4** — но только когда правило стало **принудительным артефактом**, а не прозой в списке (v1: 0/4, v2-проза: 1/4, v3-артефакт: 4/4) | 3 |
| Sonnet+метод против голой фронтир-модели (код/данные/исследования) | ничья или выигрыш в 3 из 4 | 4–5 |
| fable-judge: Haiku ловит подложенные фроды в лживом отчёте «работа завершена» | 4-и-3 из 5 → **5/5 в обоих прогонах** | 8 |
| Прирост обратно пропорционален силе модели (тезис репо) | голый Haiku 2/10 (ложное «production-ready») → с методом 6/10; Sonnet 9→10 | 12–13 |
| Сам голый Fable 5 нарушил собственное правило (деплой без спроса, предписанный README фикстуры) | 1 из 2 прогонов → отсюда родился authorization gate | 11 |
| Обычные мелкие задачи на сильных моделях | прироста НЕТ (нуль опубликован) | 1, 6, 7 |

Границы: сами авторы маркируют всё как **smoke-grade** (1–4 прогона на ячейку, LLM-судьи); слабый тир
имеет открытую проблему (молчаливо выброшенный follow-up у Haiku пережил три формулировки правила);
метод не делает знания модели свежее (голая фронтир-модель выигрывает knowledge-heavy research).

**Урок для KAIF вне зависимости от интеграции:** «слабые модели следуют правилам на точках решения, а
не правилам в списках». Это прямой ответ на выводы `researches/02` (отставание KAIF — НАДЁЖНОСТЬ
ИСПОЛНЕНИЯ: правила-как-проза не исполняются, `bugs/01`, `bugs/02`, EXP-0001) и главный переносимый
приём для **части 1 эпика** (инсталляция): инструкции установщика для слабых моделей должны требовать
принудительные артефакты-чекпоинты, а не описывать пожелания прозой.

## 5. Соотношение с KAIF — перекрытия, дополнения, конфликты

Родство глубокое: KAIF — дистилляция совместной работы Криника и Claude; fable-method — дистилляция
работы Claude Fable 5. Но **уровни разные, и это главное**:

| | KAIF | fable-method |
|---|---|---|
| Уровень | **Проект/сессии**: вынесенная память (STATUS), накопление знаний (plans/ideas/bugs/researches/EXPERIENCE), ритуалы, жизненный цикл, автономные циклы | **Одна задача**: как надёжно исполнить один запрос от классификации до отчёта |
| Что энфорсит | Дисциплину ведения проекта (документы, DONE, интервью, эскалация видения) | Дисциплину исполнения (гейты, артефакты, верификация наблюдением) |
| Верификация чужой работы | нет аналога | **fable-judge** — адверсарная, с перепрогоном |
| Не-кодовые сферы | `framework/spheres/` — библиотеки терминов | `references/domains/` — адаптеры с evidence set / фродами / авторитетами (богаче) |
| Доказательная база | нет | eval-харнесс с ловушками |

Дополнения (KAIF выигрывает от): цикл исполнения задачи + гейты + принудительные артефакты (лечит
Парето-пробел №1 из researches/02), fable-judge (у KAIF вообще нет адверсарной верификации — ни в
циклах `/dayloop`/`/nightloop`, ни в `/release`), идея trap-фикстур для проверки качества деплоя KAIF
(родня `homeworks/01`, но механизированная).

Перекрытия (нужна сшивка, а не дубль): `BUG_FIXING_FRAMEWORK.md` ↔ шаги 2/5 метода (intent gate, twin
check — у KAIF аналогов нет, стоит впитать); `PHILOSOPHY.md` (KISS/Оккам, «застрял → упрости») ↔ дух
метода — совместимы; отчёт «результатом-вперёд» ↔ KAIF «narrate»; spheres ↔ domains — пересекающиеся
концепты, надо решить судьбу.

**Конфликты, требующие явной сшивки при любом варианте интеграции:**
1. **Коммиты:** fable запрещает commit/push без явного указания юзера; git-процесс KAIF предписывает
   «коммить инкрементально и часто». Разрешение: развёрнутый KAIF = *стоячая авторизация* владельца на
   коммиты (зафиксировать это в сшивке явно; push/деплой/релиз — по-прежнему через AUTH).
2. **Интервью vs один вопрос:** fable «задай ровно один точечный вопрос»; KAIF `/interview` — до 10.
   Разные жанры (уточнение задачи vs выяснение видения) — разнести формулировками.
3. **Циклы:** fable-loop (оркестровка одной задачи субагентами) ≠ KAIF-циклы (`/autoloop` — проход по
   беклогу). Не заменять, а вложить: пункт беклога исполняется по fable-method/loop.

## 6. Варианты интеграции

### А. Независимый обновляемый модуль (плагин/скиллы рядом с KAIF)
KAIF на установке рекомендует/ставит fable-method плагином; ритуалы KAIF ссылаются на `/fable-method`,
`/fable-judge` в нужных точках.
- ➕ Ноль поддержки на нашей стороне; апстрим живой (4 релиза/10 дней) — обновления текут сами;
  доказательная база остаётся у первоисточника; ноль раздувания `KAIF.md`.
- ➖ **Плагин-механизм есть только у Claude Code** — для Codex/Grok Build/Cline/Zoo Code пришлось бы
  вручную вклеивать AGENTS.md (то есть для 4 из 5 целевых систем «модуль» всё равно вырождается в
  копирование текста); внешняя зависимость в СЕРДЦЕВИНЕ обещания KAIF (дисциплина исполнения — это
  домен KAIF, а не сторонняя утилита); bus factor 1 у апстрима; смысловой дрейф формулировок между
  версиями, KAIF-ритуалы могут ссылаться на то, что апстрим переименовал.

### Б. Хирургически в ДНК (дистилляция в собственные документы KAIF)
Впитать цикл, гейты и принудительные артефакты в `framework/*` (AGENT_GUIDE, BUG_FIXING, шаблоны
навыков), fable-judge переписать своим навыком.
- ➕ Самораспаковка сохраняется (всё в `KAIF.md`); одинаково для всех 5 систем через машинерию навыков
  KAIF; ноль внешних зависимостей (конкурентное преимущество №1 KAIF по researches/02); можно взять
  только нужное (KISS).
- ➖ Форк-дивергенция: улучшения апстрима (а они у него рождаются из **провалившихся тестов**, т.е.
  дорогие) к нам не текут; **переформулировка опасна** — формулировки метода отточены итерациями
  eval (урок v1→v3: сила именно в точной формулировке артефакта, парафраз может убить эффект);
  мы начинаем заявлять дисциплину без чеков апстрима (их eval-приёмки).

### В. Гибрид «vendor + ДНК» — **моя рекомендация**
1. **Vendor (полностью, дословно):** включить 4 навыка fable как шаблоны навыков KAIF
   (`framework/skills/fable-method|fable-loop|fable-judge|fable-domain/`) **на английском, дословно,
   с шапкой-атрибуцией** (автор, репозиторий, версия-пин, MIT). Машинерия KAIF раскатывает их во все
   5 систем как обычные навыки. `references/` брать выборочно: failure-modes + flowcharts + examples —
   да; `domains/` — решение владельца (см. §8); `eval/` — НЕ тянуть (доказательная база, не нагрузка;
   в шапке — ссылка на апстрим).
2. **ДНК (сшивка, не копия):** в документы KAIF добавить ТОЧКИ ВЫЗОВА и снять конфликты: AGENT_GUIDE
   чек-лист → «исполнение задачи — по fable-method»; BUG_FIXING → intent gate + twin check; циклы →
   пункт беклога через fable-loop, «done» пункта — через fable-judge-проход; `/release` → judge-проход
   перед публикацией; git-политика → стоячая авторизация коммитов (см. §5-конфликт-1).
3. **Обновление:** ритуал-хвост `/kaif-sync-fable` (или вручную по случаю): diff вендоренных файлов
   против апстрима, перенос дословно, версия-пин в шапке. Полуручной, редкий, дешёвый.
- ➕ Самораспаковка цела; все 5 систем в паритете; формулировки не испорчены парафразом; атрибуция
  честная; обновляемость сохранена (полуручная); KAIF получает и метод, и судью.
- ➖ `KAIF.md` растёт (~35–45K знаков = 4 навыка + 3 reference-файла; +8 доменов ≈ ещё ~30K — потому
  и вопрос владельцу); новый (лёгкий) ритуал синхронизации.

## 7. Влияние на часть 1 эпика (Thin KAIF — установка)

1. Идея 12 требует «все внутренние ИИ-документы — на английском origin» — fable уже EN-only, вендорится
   как есть, переводить нечего. Совпадение по духу полное.
2. Приём «принудительный артефакт на точке решения» — применить к САМОЙ установке: этапы инсталлятора
   для ИИ-агента (особенно финальная доменная адаптация слабой моделью) должны требовать чекпоинт-строки
   (аналог INTENT/TWINS) вместо прозы — прямое лекарство от `bugs/01`.
3. Идея trap-фикстуры: механизированная проверка «развернулся ли KAIF правильно» (расширение
   `kaif-unpack.mjs --check` до поведенческого смока) — по образу `eval/scenarios/`.
4. Вендоринг +4 навыка усиливает довод части 1: раскатка навыков в 5 систем обязана быть механической
   (шаблоны + машинерия), вручную агент будет разворачивать 26+ навыков непозволительно дорого.
