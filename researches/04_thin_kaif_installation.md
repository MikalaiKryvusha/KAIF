# Research 04 — Thin KAIF: архитектура установки с минимальными когнитивными усилиями ИИ

> Тема: переработка установки KAIF так, чтобы всю тяжесть нёс код-инсталлятор, а ИИ-агент выполнял лишь
> неустранимо-когнитивный финал. Цель: одна полноценная версия, одинаково лёгкая для сильной и слабой модели.
> Породивший документ: `ideas/12_simple_instalation+fable_method.md` (эпик KAIF 1.5 — Thin KAIF).
> Метод: разведка репозитория субагентом (карта конвейера с файлами:строками) + веб-проверка Grok Build +
> прямые замеры. Живой справочник (DONE не помечается). Создан 2026-07-16.

## 1. Диагноз: где сейчас сгорают токены и усилия

Факты по текущему конвейеру (v1.4; полная карта — в §приложении A ниже):

- `KAIF.md` = **175K знаков / 39 блоков `FILE:`**. Машинерия (`kaif-unpack.mjs`) закрывает только
  **Stage 1 (скелет verbatim) + Stage 1.5 (валидация `--check`)** и механическую трансляцию навыков
  **только для Zoo/Roo** (`--agent zoo-code|roo-code`). Всё остальное — когнитивная работа ИИ.
- **Две развилки, на которые жалуется идея 12, реальны:** (1) fast path vs respectful staged flow
  (`_intro.md:248–265`); (2) полный `KAIF.md` vs `KAIF-SLIM.md` (7.7K). Обе требуют решений от агента/юзера
  ещё ДО начала работы.
- **13 точек когнитивной нагрузки** (B1–B13). Самая тяжёлая — B1: ручной перевод ~40 файлов на язык
  владельца (проза, заголовки, `description:` навыков с триггер-фразами). Далее B2 (8 плейсхолдеров),
  B4 (сферная терминология), B5 (трансляция навыков для систем, где нет механики — а это 4 из 5 целевых),
  B6/B7 (wiring: контекст-файл, `.kaif/kaif.json`, `package.json`), B11 (скрабинг для anonymous).
- **Неустранимо-когнитивны только три:** B3 (изучение чужого проекта), B8 (seed `MASTER_PLAN.md` из
  `GOAL.md`), B9 (две карты архитектуры). Понимание проекта не шаблонизируется — и не должно.
- Первопричина всего направления — `bugs/01`: слабые локальные ЛЛМ **интерпретируют вместо копирования**,
  а Ollama молча обрезает контекст. Вывод бага и урок fable-method (researches/03 §4) совпадают:
  слабой модели нельзя давать длинную прозу с пожеланиями — только короткие механические шаги
  с принудительными чекпоинтами.

## 2. Целевая архитектура (три слоя)

```
KAIF.md (тонкий, ~300 строк)          ← единая точка входа, кладётся в проект юзера
  └─ инструкция ИИ: проверь Node → создай файл KAIF-LOADER.mjs (код дан дословно) → запусти с параметрами
       └─ KAIF-LOADER.mjs (~80 строк) ← скачивает бандл инсталлятора из origin, проверяет, запускает
            └─ KAIF-CORE.mjs + шаблоны (тяжёлая машинерия из origin-репозитория)
                 ├─ ставит ВСЁ механизируемое сам (без ИИ): доки, языки, 5 систем навыков, wiring, валидация
                 └─ в самом конце выдаёт ИИ короткое задание финальной адаптации (с чекпоинтами)
```

### 2.1 Тонкий `KAIF.md` — единая точка входа

Содержимое: краткий манифест «что такое KAIF» (для человека и агента) + **ровно одна работа для ИИ**:
1. Проверить `node -v` (см. §2.2 про Node);
2. Создать файл `KAIF-LOADER.mjs` из приведённого дословно блока `FILE:` (это уже отработанный паттерн
   Stage 1 — копирование одного файла удавалось даже слабым моделям);
3. Запустить: `node KAIF-LOADER.mjs --lang <язык> --mode <standard|anonymous>` — параметры юзер задал
   командой-инициатором, агент лишь пробрасывает их.

Исчезают обе развилки: нет fast/staged (нечего стадировать — ИИ копирует один маленький файл), нет
full/slim (версия одна, полноценная). Инструкция пишется в стиле «принудительный артефакт»: каждый из
трёх шагов заканчивается строкой-чекпоинтом, которую агент обязан показать (`KAIF-BOOT: node <версия>`,
`KAIF-BOOT: loader created sha256=...`, `KAIF-BOOT: loader exit 0`) — прямое лекарство от `bugs/01`.

### 2.2 Проблема курицы и яйца: Node

**mjs-загрузчик не может сам установить Node — ему нужен Node, чтобы запуститься.** Значит, проверка и
установка Node — единственная часть бутстрапа, которая обязана остаться на стороне ИИ-агента (у него есть
shell). Тонкий `KAIF.md` инструктирует: `node -v` → если нет, поставить через пакетный менеджер платформы
(winget/brew/apt) **спросив согласие юзера** (молчаливая установка рантайма на чужую машину — за границей
автономии KAIF; вопрос политики → интервью). Альтернатива «загрузчик на sh+ps1» отвергнута: два
платформозависимых скрипта вместо одного mjs — рост развилок, против духа идеи 12.

### 2.3 `KAIF-LOADER.mjs` — минимальный загрузчик

Один файл ~80 строк, ноль зависимостей (голый `node:https`/`fetch` + `node:fs`): скачивает из origin
бандл инсталлятора, сверяет по манифесту (имена+sha256), кладёт в `.kaif/install/`, запускает
`node .kaif/install/KAIF-CORE.mjs` с проброшенными параметрами. Источник скачивания — вопрос §3.1.

### 2.4 `KAIF-CORE.mjs` — тяжёлая машинерия-инсталлятор (работает БЕЗ ИИ)

Развитие связки `build-framework.mjs` + `kaif-unpack.mjs` (обе уже существуют и проверены — «манифест +
verbatim + `--check`»). Что делает механически:

| Шаг | Закрывает точку | Как |
|---|---|---|
| 1. Распаковка agent-only доков **на английском, verbatim** | B1 (частично) | Прямо из бандла; переводить БОЛЬШЕ НЕ НУЖНО by design |
| 2. Генерация owner-facing доков из **языковых шаблонов** (10 языков) | B1 (остаток) | `templates/languages/<lang>/...`; подстановка `{{OWNER_LANGUAGE}}` и переменных |
| 3. Автозаполнение плейсхолдеров | B2 (большая часть) | `REPO_URL`←`git remote`, `LICENSE`←файл, `BUILD/TEST`←`package.json` scripts, `PROJECT_NAME`←каталог/package.json, `AUTHOR`←`git config`, `COMMIT_COMMAND`/`AGENT` ← параметры. Незаполнимое помечается `<PLACEHOLDER>` и попадает в финальное задание ИИ |
| 4. Навыки сразу для **5 систем** | B5 | Из EN-шаблонов навыков: Claude Code `.claude/skills/`, Codex `.agents/skills/`+`AGENTS.md`, **Grok Build `.grok/skills/`** (стандарт SKILL.md подтверждён веб-разведкой 2026-07-16; Grok также сам читает `.claude/` и AGENTS.md), Cline `.cline/skills/`, Zoo Code `.roo/commands/`+`.roo/rules/` (код уже есть). Остальные системы — `AGENTS.md` по остаточному принципу |
| 5. Wiring | B6, B7 | Контекст-файлы (`CLAUDE.md`→AGENT_GUIDE и т.п.), `.kaif/kaif.json` (версия/дата/язык/mode из параметров), `kaif:*`-handles в `package.json` (если он есть) |
| 6. Anonymous by design | B11 | Автор живёт ТОЛЬКО в owner-README-шаблонах и origin-полях → `--mode anonymous` = не эмитить их + пропустить 3 origin-tied навыка (паттерн уже в `kaif-unpack.mjs`). Скрабинг прозы отмирает |
| 7. Расширенная валидация | — | `--check`-манифест покрывает: все доки, wiring, 5 систем навыков, отсутствие незаконных `{{...}}`, языковую подстановку. Цикл до 100% |
| 8. Самоочистка | B12 | Финализатор удаляет `KAIF.md`, `KAIF-LOADER.mjs`, `.kaif/install/`; остаётся только развёрнутый фреймворк + `.kaif/kaif.json` |

### 2.5 Финальное когнитивное задание ИИ (единственное)

После зелёной валидации KAIF-CORE пишет **`KAIF_ADAPTATION_TASK.md`** — короткое (1–2 экрана) задание
ровно на неустранимое (B3, B8, B9 + осиротевшие плейсхолдеры): изучи проект → заполни две карты →
выведи `MASTER_PLAN.md` из `GOAL.md` → добей помеченные `<PLACEHOLDER>` → прогони
`node .kaif/install/KAIF-CORE.mjs --verify-final`. Каждый пункт — с принудительной строкой-чекпоинтом
(`KAIF-ADAPT: maps filled`, ...), `--verify-final` механически проверяет чекпоинты и остатки
плейсхолдеров, и только затем разрешает самоочистку и коммит. Слабая модель получает короткий список
механических шагов вместо 40 файлов прозы.

## 3. Развилки проектирования (и рекомендации)

### 3.1 Откуда качать: релиз-пин vs main
- **Рекомендация: релизные ассеты, пин на версию** (`releases/download/vX.Y/kaif-core-bundle.*` +
  `manifest.json` с sha256). Стабильно, версионируемо, совместимо с ритуалом `/release` (уже прикладывает
  артефакты). `--channel main` (raw.githubusercontent.com) — опция для разработки.
- Форма бандла: либо N файлов по манифесту (проще, нет распаковки), либо один самораспаковывающийся
  `KAIF-CORE-BUNDLE.md` с блоками `FILE:` (переиспользует проверенный формат). Склоняюсь ко второму —
  один HTTP-запрос, знакомая механика, `--check` уже умеет.

### 3.2 Судьба самораспаковки (изменение канона!)
Заметка владельца в `AGENT_GUIDE.md` и Q1 интервью 001: «всё необходимое для развёртывания должно жить
внутри `KAIF.md`». Тонкий `KAIF.md` **ломает этот канон**: без сети установка невозможна.
Варианты: (а) принять смену канона («самодостаточен origin, а не файл»); (б) продолжать публиковать
полный оффлайн-артефакт в релизах (генерация уже есть — это сегодняшний `KAIF.md`) как запасной путь для
офлайн-сред; (в) считать офлайн-случай закрытым Slim'ом — но идея 12 хочет Slim убрать. **Рекомендация:
(а)+(б)**: тонкий `KAIF.md` — единственный документированный путь, полный — тихий release-ассет
«для авиарежима». → интервью.

### 3.3 Триггер-фразы навыков на языке владельца
Сейчас `description:` навыков переводится, чтобы агент ловил команды на языке владельца
(`_intro.md:66`). Идея 12 хочет навыки только на английском. Конфликт. Варианты: (а) в языковые шаблоны
включить **микро-таблицу локализованных триггер-алиасов** (одна строка на навык; машинерия дописывает её
в `description:` — навык остаётся английским, матчинг на языке владельца сохраняется);
(б) отказаться от локализованных триггеров (команды — латиницей). **Рекомендация: (а)** — дёшево,
механически, снимает конфликт. → интервью.

### 3.4 Десять языков
Предложение (по населению+IT-аудитории): **en, ru, zh-Hans, es, hi, ar, pt, fr, de, ja**. Языки вне
десятки: машинерия ставит en, финальное задание ИИ получает пункт «переведи owner-facing доки на <язык>»
(деградация к сегодняшнему поведению, но только для хвостовых языков). Кто готовит переводы шаблонов:
одноразово ИИ в репо KAIF (build-time, под ревью), НЕ при каждой установке. → интервью (состав списка).

### 3.5 Пять систем сразу — следствия
Идея 12 фиксирует: разворачиваем под все 5 систем всегда (минус одна развилка). Следствие — в проекте
появляются `.claude/ .agents/ .grok/ .cline/ .roo/` одновременно (≈ +5 каталогов). Это цена решения;
смягчение — флаг `--agents claude,zoo` для тех, кто попросит (не развилка по умолчанию, а опция).
Grok Build: адаптера в репо нет — написать по образцу существующих (формат подтверждён; при реализации
доверифицировать по docs.x.ai). Cursor/Copilot/Windsurf: их пользователей закрывает `AGENTS.md` +
чтение `.claude/skills/` (Cursor это умеет) — адаптеры остаются справочными.

### 3.6 KAIF-SLIM: убирать ли
Экономика: Slim = 0 сетевых зависимостей, 7.7K, «просто работает»; Thin = сетевой bootstrap + Node, но
полноценный фреймворк почти даром. Slim перестаёт быть нужен как «лёгкий путь», но остаётся нужен как
«офлайн/без-Node путь», ЕСЛИ не принят §3.2(б). **Рекомендация: убрать Slim в 1.5** (цель идеи),
офлайн-случай закрыть полным release-ассетом. → интервью.

## 4. Изменения в репозитории KAIF (payload-слой)

- `framework/` дополняется: `templates/languages/<lang>/` (owner-facing доки + микро-таблицы триггеров),
  `installer/` (исходники KAIF-LOADER/KAIF-CORE), манифест «universal vs owner-facing vs domain-adaptive»
  (расширение готового прото-манифеста `DOC_TARGETS` из `build-framework.mjs:61–84`).
- `build-framework.mjs` собирает ТРИ артефакта: тонкий `KAIF.md`, `KAIF-CORE-BUNDLE` (для релиза),
  полный оффлайн-`KAIF-FULL.md` (если принят §3.2(б)).
- `check-framework.mjs` учится проверять все три + прогонять smoke-установку в песочнице (см. §5).
- Ритуал `/release` прикладывает новые ассеты (канон заголовка — EXP-0004).
- Вендоренные fable-навыки (researches/03 §6-В) входят в шаблоны навыков и раскатываются той же машинерией.

## 5. Верификация эпика (как узнаем, что получилось)

1. **Песочница**: чистый временный проект → тонкий `KAIF.md` → полная установка. Мерило успеха идеи —
   **число токенов/действий ИИ** между «прочитал KAIF.md» и «валидация зелёная»: цель — на порядок меньше
   сегодняшнего (сегодня: перевод+адаптация ~40 файлов; цель: скопировать 1 файл + финальное задание).
2. **Trap-фикстура по образу fable-method** (researches/03 §7): смоделировать «слабую модель» — проверить,
   что все чекпоинт-строки выданы и `--verify-final` ловит пропуски.
3. **Полевая проверка** на реальной слабой локальной ЛЛМ в Zoo Code — переиспользовать протокол
   `homeworks/01` (и это же закроет `bugs/01`).
4. Все 5 систем: файлы навыков соответствуют официальным форматам (доверификация по докам систем на
   момент реализации — форматы «плывут», `adapters/_index.md:32–34`).

## 6. Риски

| Риск | Тяжесть | Смягчение |
|---|---|---|
| Сетевая зависимость установки (offline = нельзя) | средняя | §3.2(б): полный оффлайн-ассет в релизе |
| Автоустановка Node — инвазивно/права/кроссплатформенность | средняя | Только с согласия юзера, через пакетный менеджер платформы; политика → интервью |
| Слабая модель не справится даже с бутстрапом (3 шага) | низкая | Чекпоинт-строки + `--check` цикл; шаги мельче, чем сегодняшний Stage 1, который уже работал |
| Форматы навыков систем уплывут | средняя | Доверификация при реализации; матрица форматов в одном месте (машинерия), чинится одним патчем |
| Grok Build — новая система, адаптера нет | низкая | Формат подтверждён (SKILL.md, `.grok/skills/`); написать адаптер + механику по образцу Zoo |
| Рост поверхности валидации (не 39 блоков, а всё сгенерированное) | низкая | Манифест расширяется данными, не логикой; `--check` уже цикличен |

## Приложение A — карта текущего конвейера (разведка 2026-07-16, субагент)

Конвейер: инициатор (3 параметра: язык/система/режим, `_intro.md:42–62`) → выбор пути fast/staged
(`_intro.md:248–265`, когнитивно) → Stage 0 подтверждение цели → **Stage 1 механика**: агент создаёт
`kaif-unpack.mjs` из блока `FILE:` и запускает — 39 блоков извлекаются verbatim
(`framework/kaif-unpack.mjs:40–60`) → **Stage 1.5 механика**: `--check` до 100%
(`kaif-unpack.mjs:87–105`) → Stage 2 когнитивно: изучение проекта → `KAIF_DEPLOYMENT_PLAN.md`
(`_intro.md:305–321`) → Stage 3 когнитивно: плейсхолдеры, перевод, сфера §13, адаптер §14, wiring
(`_intro.md:323–335`) → Stage 4: `KAIF_FRAMEWORK.md`, удалить `KAIF.md`, коммит (`_intro.md:337–350`).
Anonymous: механика пропуска 3 навыков (`kaif-unpack.mjs:26,30,49,55`) + когнитивный скрабинг прозы
(`_intro.md:352–369`). Механическая трансляция навыков: только `--agent zoo-code|roo-code`
(`kaif-unpack.mjs:66–83`); Codex/Cursor/Copilot/Windsurf/Cline — прозаические адаптеры (делает ИИ).
Плейсхолдеров deploy-time — 8 канонических (`_intro.md:409–425`) + разрозненные (67 вхождений в 17
файлах `framework/`). Прото-манифест адаптируемости уже есть: note-поля `DOC_TARGETS`
(`build-framework.mjs:61–84`) — «write verbatim» / «replace every placeholder» / «seed» / «derive».
